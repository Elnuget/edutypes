# RM Universal Para Crear Cursos En Otros Repositorios

## Proposito
Este documento sirve para crear cursos de cualquier tema en cualquier repositorio, incluso si no tiene relacion con programacion ni con este proyecto.

Debe funcionar como un manual maestro para que un agente pueda tomar:
- el tema del curso
- el tipo de estudiante
- el resultado esperado

y a partir de eso construir un curso completo con buena estructura, buena pedagogia, buena experiencia de uso y buena logica de evaluacion.

Ejemplos de cursos que este RM debe poder cubrir:
- ecuaciones diferenciales
- guitarra
- algebra lineal
- historia del arte
- ingles conversacional
- marketing digital
- fotografia
- ajedrez
- programacion

Este RM no depende de una tecnologia concreta. Se puede aplicar en:
- apps web
- apps moviles
- plataformas educativas
- proyectos React
- proyectos Vue
- sitios estaticos
- LMS personalizados
- repositorios nuevos desde cero

## Objetivo Del RM
Que cualquier agente pueda replicar un curso bien diseñado aunque cambie:
- el tema
- la tecnologia
- el repositorio
- el publico

Lo que no debe cambiar es la calidad del diseño instruccional.

## Principio Central
Un curso bueno no es solo una lista de temas. Es una ruta de transformacion.

Por eso, antes de crear contenido, el agente debe definir:
- quien entra al curso
- con que nivel entra
- que sabe al terminar
- que puede hacer al terminar
- como se demuestra ese aprendizaje

## Marco Base Para Diseñar Cualquier Curso

### 1. Punto De Entrada
El agente debe definir con precision:
- edad aproximada o tipo de estudiante
- nivel inicial real
- errores tipicos del principiante
- vocabulario que entiende
- frustraciones comunes
- velocidad de aprendizaje esperada

### 2. Punto De Salida
El curso debe cerrar con capacidades verificables.

No basta con decir:
- “entendera el tema”

Hay que decir:
- “podra resolver ecuaciones diferenciales separables”
- “podra tocar cambios basicos de acordes con ritmo estable”
- “podra identificar y aplicar el concepto en un ejercicio real”

### 3. Unidades De Transformacion
Cada unidad debe cambiar algo concreto en el estudiante.

Una unidad no debe ser solo un titulo tematico. Debe tener:
- foco
- objetivo
- habilidades concretas
- errores a corregir
- criterio de cierre

### 4. Escalera De Aprendizaje
Todo curso debe crecer por escalones.

Orden recomendado:
1. reconocer la pieza
2. nombrar la pieza
3. entender para que sirve
4. ver su forma o estructura
5. usarla de forma aislada
6. combinarla con otra
7. aplicarla en un caso real
8. consolidarla con repeticion guiada

Este orden aplica tanto para:
- una formula matematica
- una tecnica musical
- una regla gramatical
- una herramienta de software

## Filosofia Pedagogica Universal

### No Suponer Lo No Enseñado
Nunca pedir que el estudiante haga algo que aun no se ha desarmado antes.

Ejemplos:
- no pedir resolver una ecuacion completa antes de explicar que representa cada termino
- no pedir un rasgueo con cambios de acordes antes de explicar postura, mano y tiempo
- no pedir un analisis textual antes de explicar que mirar en el texto

### Una Dificultad A La Vez
Cada paso debe atacar una sola dificultad principal.

Si una actividad exige muchas cosas al mismo tiempo, el estudiante no sabe que corregir.

### Explicacion Corta, Practica Repetida
Es mejor:
- una explicacion corta
- seguida por varias practicas pequeñas

que:
- una explicacion larga
- seguida por una sola practica grande

### Aprendizaje Visible
Siempre que el tema lo permita, el estudiante debe ver un resultado.

Ejemplos:
- en matematicas: el resultado de la ecuacion, la grafica o el procedimiento correcto
- en guitarra: el patron, el acorde, el ritmo o una guia de sonido/video
- en idiomas: frase producida, correccion y comparacion
- en programacion: consola, UI o salida del codigo

## Estructura Pedagogica Recomendada

### Nivel 1. Curso
El curso debe definir:
- promesa principal
- nivel de entrada
- resultado final
- numero estimado de unidades
- criterio de dominio

### Nivel 2. Unidad
Cada unidad debe incluir:
- nombre
- objetivo
- que problema resuelve
- que habilidad desarrolla
- prerequisitos
- entregable o criterio de cierre

### Nivel 3. Leccion
Cada leccion debe incluir:
- una meta pequeña
- explicacion minima
- uno o varios ejemplos
- practica guiada
- practica de consolidacion
- feedback

### Nivel 4. Micro-paso
La leccion debe dividirse en micro-pasos.

El modelo recomendado es:
- tarjeta breve
- ejercicio
- ejercicio
- tarjeta breve
- ejercicio
- ejercicio

Ese ritmo no es obligatorio en todos los proyectos, pero es una base muy buena para cursos donde se busca:
- consumo rapido
- poca friccion
- mucha practica
- progresion clara

## Como Explicar Bien Cualquier Tema

### Regla De Los Cuatro Niveles
Cuando aparezca una pieza nueva, explicarla en este orden:
1. que es
2. para que sirve
3. como se ve
4. como se usa

### Regla De Descomposicion
Si algo tiene partes, explicar cada parte.

Ejemplos:

En matematicas:
- que significa cada simbolo
- que representa cada termino
- donde se suma, resta o deriva

En musica:
- donde va cada dedo
- que cuerda se toca
- en que tiempo cae el cambio

En idiomas:
- que funcion tiene cada palabra
- cual es el orden
- por que una opcion es correcta y otra no

### Regla De Lenguaje Simple
- frases cortas
- una idea por frase
- sin adornos innecesarios
- sin tecnicismos antes de tiempo
- si aparece un tecnicismo, explicarlo de inmediato

## Diseño De Ejercicios

### El Ejercicio Debe Ser Resolvible
El estudiante no debe perder tiempo adivinando que se esperaba.

La instruccion debe decir:
- que hacer
- con que material
- con que formato
- que se considera correcto

### Tipos De Ejercicios Segun La Materia

#### Materias Exactas
Ejemplos:
- completar pasos
- despejar una expresion
- elegir el metodo correcto
- resolver un caso
- justificar cada paso

#### Materias Practicas Motoras
Ejemplos:
- repetir una postura
- tocar un patron
- cambiar entre dos posiciones
- seguir una cuenta
- grabar o marcar ejecucion

#### Materias Conceptuales
Ejemplos:
- identificar
- clasificar
- comparar
- explicar con tus palabras
- aplicar a un caso

#### Programacion O Logica Formal
Ejemplos:
- escribir una estructura
- corregir un error
- completar una funcion
- predecir una salida
- producir un resultado visible

### Cantidad De Ejercicios
Es mejor tener muchos ejercicios pequeños que pocos ejercicios gigantes.

La practica debe ser:
- frecuente
- corta
- guiada
- acumulativa

## Feedback Y Correccion

### Regla General
El feedback debe ayudar a corregir rapido.

No debe obligar al estudiante a pensar demasiado en “que quiso decir la plataforma”.

### Como Debe Ser El Feedback
- claro
- breve
- concreto
- accionable
- honesto
- ubicado en el error real

### Estructura Recomendada Del Feedback
1. que esta bien
2. que esta mal
3. que debe corregir exactamente
4. si puede, donde esta el error

### Ejemplos De Buen Feedback

Matematicas:
- “El metodo elegido si sirve, pero en el segundo paso cambiaste mal el signo de `-3` a `+3`.”
- “La derivada esta bien planteada. Lo que falta es integrar la parte derecha.”

Guitarra:
- “La forma del acorde va bien, pero la tercera cuerda queda apagada.”
- “El cambio esta correcto. Lo que falta es mantener el pulso constante en 4 tiempos.”

Idiomas:
- “La idea es correcta, pero el verbo debe ir en pasado.”
- “La palabra elegida se entiende, pero aqui se necesita una preposicion distinta.”

Programacion:
- “Despues de `student` te falta `:`.”
- “Escribiste `unloked`, pero debe ser `unlocked`.”

### Evitar Feedback Vago
No usar:
- “incorrecto”
- “revisa”
- “hay un error”

salvo que no exista forma realista de ser mas preciso.

## Mensajes De Exito
El mensaje de exito debe reforzar la motivacion.

Debe:
- felicitar
- confirmar lo que salio bien
- marcar progreso
- si aplica, mostrar resultado

Ejemplos:
- “Muy bien. Resolviste correctamente el procedimiento.”
- “Excelente. El cambio de acordes ya sale limpio.”
- “Bien hecho. Tu respuesta usa la estructura correcta.”

## Uso De Resultados Visibles
Cuando el tema lo permita, mostrar un resultado visible mejora mucho la experiencia.

Ejemplos:
- grafica
- solucion final
- comparacion entre respuesta y modelo
- reproduccion de audio o ritmo
- consola o salida de codigo
- vista previa de una composicion o ejercicio

El resultado visible debe:
- ser facil de entender
- conectar con la accion del estudiante
- aparecer justo despues de la revision cuando sea posible

## Diseño De UX Para Plataformas De Curso

### Regla Principal
La interfaz no debe competir con el aprendizaje.

### Landing O Pantalla Inicial
Debe mostrar:
- de que trata el curso
- para quien es
- que logra al terminar
- por donde empezar
- progreso si ya existe

### Pantalla De Leccion
Debe privilegiar:
- foco
- claridad
- poco texto a la vez
- una accion principal

### Scroll
Evitar paginas larguisimas si el producto busca sesiones cortas.

Si el curso esta pensado para:
- leer un rato
- avanzar por pasos
- volver luego

entonces conviene:
- flujo por tarjetas
- navegacion por pasos
- progreso persistente

### Responsive
Disenar primero para movil si el uso esperado es frecuente en telefono.

Preguntas obligatorias:
- se puede leer sin zoom
- se puede tocar sin errores
- el contenido se desborda
- el editor o area de entrada sigue siendo usable
- la correccion sigue visible sin saturar la pantalla

## Persistencia Y Progreso

### Progreso Minimo Recomendado
Guardar:
- unidad actual
- leccion actual
- paso actual
- ejercicios completados
- respuestas del estudiante si aplica

### Tipos De Progreso
- secuencial: una pieza desbloquea la siguiente
- libre con recomendacion: todo visible, pero con orden sugerido
- mixto: unidades abiertas por bloques

Para cursos fundacionales, suele funcionar mejor el progreso secuencial.

### Reinicio
Si existe boton de reinicio:
- debe pedir confirmacion
- debe explicar que se va a borrar

## Evaluacion Del Curso

### Evaluacion De Micro-logros
Cada ejercicio debe medir una pieza pequena.

### Evaluacion De Cierre
Cada unidad debe tener una forma de confirmar dominio.

Ejemplos:
- mini proyecto
- practica integrada
- problema completo
- interpretacion o ejecucion completa
- ensayo corto

### Evaluacion Final Del Curso
El cierre del curso debe demostrar que el estudiante puede usar lo aprendido en un caso real.

## Adaptacion Segun Tipo De Curso

### Si El Curso Es Muy Tecnico
Priorizar:
- precision
- pasos pequeños
- ejemplos reales
- validacion estricta

### Si El Curso Es Practico-Corporal
Priorizar:
- demostraciones
- repeticiones cortas
- ritmo
- checkpoints visuales o auditivos

### Si El Curso Es Teorico
Priorizar:
- claridad conceptual
- comparaciones
- mapas mentales
- preguntas de aplicacion

### Si El Curso Es Creativo
Priorizar:
- imitacion guiada
- exploracion controlada
- criterios concretos de mejora
- ejemplos buenos y malos

## Arquitectura Recomendada Para Otro Repositorio
Si el agente crea un curso en un repo nuevo, esta es una estructura razonable:

- `docs/`
- `src/`
- `src/data/`
- `src/components/`
- `src/lib/`
- `src/styles/`
- `src/content/`

### Distribucion Sugerida
- `src/data/` para unidades, lecciones y metadatos
- `src/components/` para pantallas y widgets de aprendizaje
- `src/lib/` para progreso, evaluacion y validacion
- `src/content/` para contenido extenso o assets de apoyo
- `docs/` para manuales, reglas del curso y prompts de replicacion

## Proceso Que Debe Seguir El Agente

### Fase 1. Definicion
- entender el tema
- entender al estudiante
- definir el resultado
- decidir tipo de progreso

### Fase 2. Arquitectura Del Curso
- dividir en unidades
- dividir cada unidad en lecciones
- dividir cada leccion en micro-pasos

### Fase 3. Contenido
- escribir tarjetas cortas
- escribir ejemplos
- escribir ejercicios
- escribir feedback esperado

### Fase 4. Producto
- diseñar pantallas
- integrar progreso
- integrar validacion
- integrar mensajes de exito y error

### Fase 5. Verificacion
- revisar orden ascendente
- revisar claridad
- revisar ambiguedades
- revisar responsive
- revisar persistencia
- revisar evaluacion

## Checklist Universal Antes De Dar Un Curso Por Terminado
- el resultado final del curso es claro
- el estudiante ideal esta definido
- las unidades suben de forma ascendente
- no hay saltos bruscos
- cada concepto nuevo se explica antes de usarse
- hay mucha practica
- las instrucciones son resolubles
- el feedback es preciso
- el exito se celebra
- el progreso se guarda
- la experiencia movil es usable
- la interfaz no distrae
- el curso tiene una prueba final o cierre integrador

## Prompt Universal Para Encargar Un Curso A Otro Agente

```text
Quiero que crees un curso completo en este repositorio.

Tema del curso: [AQUI EL TEMA]
Tipo de estudiante: [AQUI EL PERFIL]
Nivel inicial: [AQUI EL NIVEL]
Resultado final esperado: [AQUI EL RESULTADO]
Tipo de experiencia deseada: [web, movil, tarjetas, clases largas, etc.]

Debes seguir obligatoriamente el archivo UNIVERSAL_COURSE_RM.md.

Reglas clave:
- construir una ruta de aprendizaje ascendente
- no asumir conocimiento no enseñado
- dividir el contenido en micro-pasos
- escribir explicaciones claras y simples
- incluir practica frecuente
- dar feedback preciso y accionable
- diseñar la experiencia para que el estudiante siempre sepa que hacer despues
- adaptar la validacion y la evidencia al tipo de materia
- mantener coherencia entre contenido, interfaz y evaluacion
```

## Criterio Final De Calidad
Un curso nuevo esta bien hecho solo si logra estas cinco cosas:
- el estudiante entiende que esta aprendiendo y por que
- cada paso se siente alcanzable
- la practica realmente consolida la teoria
- el feedback permite corregirse rapido
- el producto final se siente coherente con la materia y el publico

Si falla una de esas cinco, el curso todavia necesita trabajo.
